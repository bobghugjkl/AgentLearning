## 七、避坑指南

### 当前大模型方向的常见误区

**误区 1：只会调 API，所有"项目"都是 OpenAI 套壳 + 几行 Streamlit**

这是最致命的。面试官 30 秒就能看出你的项目是教程跟练。**判断标准**：你的项目里有没有非显而易见的技术决策？比如"为什么 chunk_size 选 400 不选 800"——你能用具体的实验数据回答，而不是"经验值"。

**误区 2：技术栈广而不深**

LangChain 全家桶 + LangGraph + 向量库 + 微调 + 推理优化全都浅尝辄止。**应届生最好是「LangChain/LangGraph 应用层 80 分 + LLM 底层 60 分 + 一个领域纵深（CV/医疗）」的 T 字结构**，不要追求 M 字。

**误区 3：把 markdown 笔记当作能力**

你 markdown 里有大量"面试考点"——但**写在笔记里 ≠ 能在压力下脱口而出**。每周必须做 1 次模拟面试（对镜子讲、录音回放、找同学互问）。

**误区 4：过度迷信「教程项目数」**

5 个浅项目的简历竞争力 < 2 个深项目。**优先把 2 个项目做到极致**（前端、Docker、量化数据、博客、Demo 部署），有余力再做第 3 个。

**误区 5：不投实习只埋头学**

学到完美状态再投——这个状态永远不会到。日常实习投递本身就是最好的能力反馈，被刷几次知道哪里不行，比闷头学三个月精准得多。

### 面试中容易被刷掉的原因

**1. 项目讲不出量化数据**
"你的 RAG 准召多少？" → "我没测过" → 凉
**对策**：每个项目至少 3 个数字，写在笔记本上每天背。

**2. 项目讲不出取舍**
"你为什么选 Milvus 不选 pgvector？" → "Milvus 比较快" → 凉
**对策**：每个技术决策能讲清「另一个方案是什么 / 为什么没选 / 选这个的代价是什么」。

**3. LangGraph 原理回答模糊**
"add_messages reducer 解决什么问题？" → "合并消息" → 平庸
**对策**："并行节点写入 messages 字段时，默认覆盖会丢失数据，add_messages 作为 reducer 把多路写入按时间序列合并到 list，是 LangGraph 支持并行/分支的关键设计"——这种结构化回答。

**4. 八股完全靠背**
"vLLM 的 PagedAttention 怎么工作？" → 背了一段但讲不出"为什么需要分页" → 凉
**对策**：每个八股至少自己跑过一次（vLLM 起服务、KV Cache 实测、LoRA 训练）。

**5. 没有代码 Demo**
"项目能给我看一下吗？" → "我本地跑的" → 信任度大降
**对策**：所有项目部署到公网（Vercel/Railway/Render 免费层），或至少 GitHub README 带 GIF 演示。

### 如何避免「看起来学了很多但没有竞争力」

最常见的失败模式：5 周学完 LangChain 全家桶 + 5 周学 RAG + 5 周学 Agent，但每周只是写了 200 行 demo 代码，没有一个项目超过 1000 行真实业务逻辑。

**反向自检表**（每周日问自己）：

```
□ 这周写的代码合并起来超过 500 行了吗？
□ 这周的代码涉及至少 3 个组件协作（不是单文件 demo）吗？
□ 这周的代码部署到公网/服务器跑了吗？
□ 这周看的论文我能口头复述核心思想吗？
□ 这周遇到的 bug 我写进 lessons learned 文档了吗？
□ 这周和别人讨论过我的项目吗（哪怕在群里）？

勾选 < 4 项 → 警告：进入「学习剧场」状态
```
